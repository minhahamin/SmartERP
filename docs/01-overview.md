# 1. 프로젝트 개요

## 1.1 프로젝트 한눈에 보기

| 항목 | 내용 |
|---|---|
| 프로젝트명 | **ERPilot** (ERP + Copilot) |
| 한 줄 정의 | ERP 데이터와 사내 지식을 AI가 대신 찾아주는 **AI-Native B2B SaaS ERP** |
| 타깃 고객 | 50~300인 규모의 제조/유통/서비스 중소기업 |
| 핵심 가치 | "엑셀과 사내 게시판에 흩어진 업무 데이터를, 누구나 말로 물어보면 바로 찾아주는 ERP" |
| 제공 형태 | Multi-tenant SaaS (회사 단위 워크스페이스 분리) |

## 1.2 프로젝트 목표

1. **운영 가능한 수준의 ERP 코어 구현**
   - 인사·재무·영업·생산·재고를 아우르는 11개 핵심 도메인을 CRUD가 아니라 실제 업무 프로세스(승인, 상태 전이, 권한 분기) 단위로 설계·구현한다.
   - "장난감 프로젝트"가 아니라 실제 중소기업 ERP 도입 컨설팅에서 요구되는 최소 기능 범위(MVP)를 충족한다.

2. **사내 업무 도우미로서의 AI 적용**
   - 단순 챗봇이 아니라 **Function Calling 기반 ERP 데이터 조회**와 **RAG 기반 사내 문서 검색**을 결합한 하이브리드 AI 어시스턴트를 구현한다.
   - AI 응답에도 사용자의 RBAC 권한이 동일하게 적용되어, "AI를 통해 권한 우회 조회가 불가능"하다는 것을 핵심 기술 포인트로 증명한다.

3. **실무 수준의 엔지니어링 역량 증명**
   - 인증/인가, 데이터 모델링, API 설계, 벡터 검색, 배포 자동화까지 백엔드/프론트엔드/AI/인프라 전 영역을 포트폴리오 한 프로젝트 안에서 입증한다.

## 1.3 해결하려는 문제

중소·중견기업 현장에서 반복적으로 발생하는 다음 문제들을 ERPilot이 해결한다.

| 문제 상황 | 기존 방식의 한계 | ERPilot의 해결 방식 |
|---|---|---|
| "재고가 얼마나 남았지?" | 담당자에게 메신저로 물어보거나 엑셀을 열어 직접 필터링 | AI 챗봇에 자연어로 질문 → 실시간 DB 조회 결과를 표/문장으로 즉답 |
| "연차 규정이 어떻게 되더라?" | 사내 위키/공유 드라이브에서 PDF를 찾아 Ctrl+F | RAG 기반 문서 검색으로 관련 조항만 발췌·요약하여 답변 |
| 신입사원 온보딩 시 반복 질문 | HR팀이 같은 질문에 매번 응답 | 반복 질문 패턴을 분석해 FAQ를 AI가 자동 생성·축적 |
| ERP 화면이 많고 복잡해 비전문 직원의 접근성이 낮음 | 메뉴를 외워야 하고, 보고서용 화면을 따로 찾아야 함 | 챗봇 대화 한 번으로 보고서성 데이터(매출 요약, 근태 현황)를 즉시 획득 |
| 부서별 권한 관리가 엑셀/구두 합의로 운영됨 | 권한 변경 시 개발자 호출, 추적 불가 | 권한 관리 화면에서 역할(Role) 단위로 메뉴/액션 권한을 실시간 토글 |

## 1.4 기대 효과

- **업무 조회 시간 단축**: 단순 조회성 업무(재고 확인, 매출 요약, 근태 확인)를 "찾는 시간"에서 "물어보는 시간"으로 전환 → 체감 처리 시간 단축
- **신규 입사자 온보딩 비용 절감**: 사내 규정/매뉴얼을 AI가 1차 응대 → HR/관리자의 반복 응대 부담 감소
- **데이터 기반 의사결정 보조**: 통계 분석 대시보드 + AI 자연어 질의로 비개발/비분석 직군도 즉시 인사이트 확인 가능
- **감사 추적성(Auditability) 확보**: 모든 AI 조회와 데이터 변경 이력이 로그로 남아, 내부통제/감사 대응에 활용 가능

## 1.5 차별화 포인트

### (1) "조회 전용 AI"가 아니라 "권한을 인지하는 AI"
대부분의 ERP+AI 포트폴리오는 AI에게 DB 풀 액세스를 주고 텍스트를 SQL로 변환하는 수준에서 멈춘다. ERPilot은 **로그인한 사용자의 JWT 클레임(Role, Department, UserId)을 Function Calling의 인자 제약 조건으로 강제 주입**하여, 일반 직원이 "전체 직원 연봉 평균"을 물어도 본인이 조회 권한이 없는 데이터는 절대 반환되지 않도록 설계했다. 이는 실무 ERP 도입 시 가장 먼저 막히는 "AI 보안 거버넌스" 문제를 정면으로 다룬다.

### (2) ERP 데이터 조회 + 문서 검색의 하이브리드 RAG
"이번 달 매출 알려줘"는 정형 데이터(SQL) 질의가 필요하고, "연차 규정 알려줘"는 비정형 문서(RAG) 검색이 필요하다. ERPilot은 LLM이 질문 의도를 분류(Intent Routing)하여 두 경로(Structured Query / Unstructured RAG)를 자동으로 선택하고, 필요한 경우 두 결과를 조합해 답변하는 구조를 갖는다.

### (3) 실무형 데이터 모델
단순 게시판형 CRUD가 아니라 상태 전이(주문 상태, 생산 상태, 휴가 승인 상태)와 트랜잭션 무결성(재고 입출고 시 동시성 처리)을 고려한 스키마를 설계하여, 실제 서비스 운영 환경에서 발생하는 데이터 정합성 이슈를 다룬다.

### (4) SaaS 관점의 설계
단일 회사용 내부 시스템이 아니라, 여러 회사가 입주할 수 있는 멀티테넌시 구조(부서/권한 체계가 회사마다 다를 수 있음을 고려한 Role-Permission 매핑 테이블)와 배포 자동화(CI/CD)를 포함하여 "SaaS 제품을 만들고 운영할 수 있는 엔지니어"임을 보여준다.

## 1.6 포트폴리오로서의 포지셔닝

| 평가 축 | 증명 포인트 |
|---|---|
| 백엔드 역량 | NestJS 모듈 설계, Prisma 트랜잭션, JWT/RBAC 보안, RESTful API 설계 원칙 |
| 프론트엔드 역량 | React Query를 통한 서버 상태 관리, Zustand로 클라이언트 상태 분리, shadcn/ui 기반 디자인 시스템 구축 |
| AI 엔지니어링 역량 | LangChain Function Calling, RAG 파이프라인(Chunking-Embedding-Retrieval), pgvector 운영 |
| 인프라 역량 | Docker 멀티 컨테이너 구성, Nginx 리버스 프록시, GitHub Actions CI/CD, AWS EC2 배포 |
| 제품 설계 역량 | 사용자 정의, 권한 모델링, 화면 흐름 설계 등 기획자 관점의 의사결정 |
