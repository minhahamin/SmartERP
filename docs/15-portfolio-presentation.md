# 15. 포트폴리오 발표 자료

## 15.1 30초 엘리베이터 피치

> "ERPilot은 ERP와 AI 챗봇을 결합한 SaaS입니다. 단순히 ChatGPT를 ERP에 붙인 게 아니라, **로그인한 사용자의 권한을 AI의 도구 호출 단계에서 서버가 강제로 재검증**하도록 설계해서, AI를 통해서도 권한 밖의 데이터(타인 급여, 타 부서 정보)는 절대 조회되지 않는 걸 핵심 기술 포인트로 잡았습니다. 백엔드는 NestJS+PostgreSQL, AI는 OpenAI Function Calling과 pgvector 기반 RAG로 구현했고, Docker/Nginx/GitHub Actions로 AWS EC2에 직접 배포까지 마쳤습니다."

## 15.2 기술적 난이도 — 강조할 3가지 포인트

### ① AI 계층에서의 권한 격리(가장 차별화되는 부분)
- **문제**: LLM에게 DB 조회 도구를 주면, 사용자가 "전체 직원 연봉 평균 알려줘"처럼 권한 밖 요청을 했을 때 LLM이 임의로 `userId`를 비워 전체 조회를 시도할 수 있다.
- **해결**: Function Calling 인자를 신뢰하지 않고, 도구 실행 직전 서버가 호출자의 `role/userId/companyId`로 인자를 강제 재작성하는 `withRbacScope` 래퍼를 모든 도구에 일괄 적용([09-ai-chatbot-design.md](09-ai-chatbot-design.md) 9.3). 추가로 권한이 없는 도구는 LLM에게 노출 목록 자체에서 제거하는 이중 방어.
- **왜 어려운가**: "프롬프트로 못하게 지시"하는 수준이 아니라, **프롬프트 인젝션에도 깨지지 않는 시스템 레벨 강제**가 필요했다는 점이 일반적인 ChatGPT 연동 튜토리얼과의 차이.

### ② 재고/입출고의 동시성 정합성
- **문제**: 동시에 여러 출고 요청이 들어오면 재고가 음수가 되거나(Race Condition), 입출고 이력과 현재 수량이 어긋날 수 있다.
- **해결**: `StockMovement` insert + `Inventory` upsert를 하나의 Prisma `$transaction`으로 묶고, 동시 요청에 대해 `SELECT ... FOR UPDATE` 비관적 락으로 직렬화([07-database-design.md](07-database-design.md) 7.6-4).

### ③ RAG 검색 품질과 권한 가시성의 동시 충족
- **문제**: 벡터 유사도만으로 검색하면 "유사하지만 내가 봐서는 안 되는 문서"가 검색될 위험이 있다.
- **해결**: pgvector 유사도 검색 SQL의 `WHERE` 절에 `company_id`, `is_public/department_id` 조건을 함께 포함시켜, **검색 후 필터링이 아니라 DB 레벨에서부터 권한 범위 내에서만 검색**되도록 설계([11-pgvector-design.md](11-pgvector-design.md) 11.5).

## 15.3 문제 해결 과정 (STAR 포맷 사례)

### 사례 1 — AI 권한 우회 문제
- **S(상황)**: 초기 설계에서는 LLM이 호출하는 함수 인자를 그대로 믿고 Prisma 쿼리를 실행했다.
- **T(과제)**: 테스트 중 일반 직원 계정으로 "팀장 급여 알려줘"라고 질문했을 때, LLM이 임의의 이름을 인자로 넘기면 다른 사람의 급여가 조회되는 시나리오를 발견했다.
- **A(행동)**: 모든 도구 실행을 `withRbacScope` 공통 래퍼로 감싸 LLM이 보낸 인자를 서버가 호출자의 실제 권한으로 덮어쓰도록 리팩터링했고, RolePermission 조회로 도구 노출 자체를 1차 필터링했다.
- **R(결과)**: 권한 밖 데이터 요청 시 AI가 "본인 데이터만 조회할 수 있습니다"로 응답하도록 일관되게 동작하게 되었고, 이 설계를 README/발표자료의 핵심 차별점으로 삼았다.

### 사례 2 — 벡터 검색 결과가 기대와 다른 문제
- **S**: "신입사원 연차 규정"으로 검색했는데 관련도가 낮은 문서가 상위에 노출되는 문제가 있었다.
- **T**: Chunk 크기가 너무 커서(2000 토큰) 하나의 Chunk에 여러 주제가 섞여 임베딩 벡터가 특정 의미에 집중되지 못했다.
- **A**: Chunk 크기를 500토큰/overlap 50으로 줄이고, RecursiveCharacterTextSplitter의 분리자 우선순위를 문단→문장 단위로 조정했다.
- **R**: 동일 질의에 대한 상위 결과의 코사인 유사도가 0.4대에서 0.7대로 개선되었고, 임계치(0.6) 미달 시 "결과 없음"을 명확히 반환하도록 만들어 잘못된 답변(hallucination) 가능성을 낮췄다.

## 15.4 AI 적용 이유

- **단순 자동화가 아니라 "접근성" 문제 해결**: 중소기업 ERP의 실제 문제는 기능 부족이 아니라 "어디에 뭐가 있는지 모름"이다. 챗봇 인터페이스는 메뉴 구조를 외울 필요 없이 자연어로 바로 원하는 데이터/문서에 도달하게 한다.
- **정형 데이터(SQL)와 비정형 데이터(문서)를 한 인터페이스로 통합**: "매출 요약"과 "연차 규정"은 본질적으로 다른 시스템(DB 조회 vs 문서 검색)이 필요한데, 사용자 입장에선 그 구분이 무의미하다. Function Calling 기반 라우팅으로 이 구분을 시스템이 대신 처리한다.
- **보안을 타협하지 않는 AI 도입이 실무에서 가장 중요한 허들**이라고 판단했다. 실제로 기업이 AI ERP 도입을 주저하는 핵심 이유가 "데이터 유출/권한 우회 우려"이므로, 이를 정면으로 설계에 반영한 것이 이 프로젝트의 실무적 설득력을 높이는 지점이다.

## 15.5 성능 최적화

| 항목 | 적용 내용 | 효과 |
|---|---|---|
| 1차 라우팅 분류에 저비용 모델 사용 | gpt-4o-mini로 ERP_DATA/DOCUMENT_SEARCH/HYBRID/GENERAL 분류 후 도구 셋 축소 | 메인 모델에 전달하는 토큰 수 감소, 응답 지연 단축 |
| SSE 스트리밍 응답 | 토큰 생성 즉시 전송(Nginx `proxy_buffering off`) | 사용자 체감 응답 시간(첫 토큰까지의 시간) 단축 |
| pgvector HNSW 인덱스 | `vector_cosine_ops` + `m=16, ef_construction=64` | Top-K 검색을 풀스캔 대비 빠르게 처리 |
| React Query 캐싱 | 목록/상세 데이터 staleTime 설정, Optimistic Update | 동일 데이터 재조회 네트워크 비용 감소, UI 반응성 개선 |
| FAQ 캐시 우선 매칭 | 반복 질문은 Function Calling/RAG 생략하고 캐시된 답변 우선 반환 | 반복 질의의 OpenAI API 호출 비용/지연 절감 |
| 문서 요약 캐시(Redis) | `summary:{documentId}:{version}` 키로 Map-Reduce 요약 결과 캐시 | 동일 문서 재요약 요청 시 LLM 재호출 없이 즉시 응답 |
| DB 인덱스 설계 | `unique(productId, warehouseId)`, `index(status, dueDate)` 등 | 재고/생산 지연 조회처럼 AI가 빈번히 호출하는 쿼리의 응답 속도 확보 |

## 15.6 보안 고려사항 (발표 시 강조 포인트)

1. **다층 권한 검증**: 화면(Route Guard) → API(PermissionsGuard) → AI 도구 실행(withRbacScope) 3중 강제, 어느 한 경로로도 우회 불가
2. **토큰 보안**: Access Token은 메모리만 사용(휘발), Refresh Token은 httpOnly+SameSite=Strict 쿠키 + Rotation([12-jwt-auth-design.md](12-jwt-auth-design.md))
3. **멀티테넌시 격리**: 모든 쿼리에 `companyId` 강제 필터, 장기적으로 Postgres RLS로 2차 방어
4. **AI 특화 보안**: 문서 Chunk 내 텍스트를 "지시"가 아닌 "데이터"로 격리하여 프롬프트 인젝션 방어, 도구 호출 한도(턴당 4회)로 비용/오작동 가드
5. **감사 추적성**: 급여 확정, 권한 변경, 로그인 실패 등 민감 액션은 `AuditLog`에 기록하여 사후 추적 가능

## 15.7 예상 질문 & 답변

**Q. 왜 별도 벡터 DB(Pinecone 등) 대신 pgvector를 선택했나요?**
> A. 문서 메타데이터(회사/부서/공개여부)와 벡터를 같은 트랜잭션, 같은 DB에서 다룰 수 있어 가시성 필터를 SQL `WHERE`절 하나로 처리할 수 있었습니다. 별도 Vector DB를 쓰면 권한 필터링을 애플리케이션 레이어에서 별도로 동기화해야 하는데, 이 경우 동기화 누락이 곧 보안 취약점이 됩니다. 현재 규모(회사당 수만 Chunk)에서는 pgvector의 HNSW 인덱스로 충분한 성능이 나온다고 판단했습니다.

**Q. LLM이 잘못된 함수를 호출하거나 인자를 잘못 생성하면 어떻게 되나요?**
> A. 두 가지로 방어합니다. 첫째, 권한이 없는 도구는 목록 자체에서 제외되어 호출 시도조차 할 수 없습니다. 둘째, 잘못된 인자(예: 다른 사용자 ID)가 들어와도 서버가 실행 직전에 호출자의 실제 권한 범위로 인자를 덮어쓰기 때문에, LLM의 실수가 실제 데이터 유출로 이어지지 않습니다.

**Q. 토큰 비용은 어떻게 관리했나요?**
> A. 라우팅 분류에 저비용 모델을 쓰고, 세션 컨텍스트를 최근 10턴으로 제한했습니다. 반복적으로 들어오는 질문은 FAQ 자동 생성 파이프라인을 거쳐 캐시된 답변으로 우선 응답하도록 해서 동일 질문에 대한 반복 호출 비용을 줄였습니다.

**Q. 멀티테넌시는 실제로 어떻게 검증했나요?**
> A. 모든 1차 마스터 테이블에 companyId를 두고 서비스 레이어에서 모든 조회에 강제 포함시켰습니다. 추가로 Postgres RLS를 적용해 애플리케이션 코드에 버그가 있어도 DB 레벨에서 다른 회사 데이터가 조회되지 않도록 2중 방어를 설계했습니다.

**Q. 이 프로젝트에서 가장 어려웠던 부분은?**
> A. 기능을 만드는 것보다, "편의성(AI가 뭐든 답해주는 것)"과 "보안(권한 밖 데이터는 절대 안 됨)" 사이의 균형을 맞추는 설계가 가장 어려웠습니다. 실무에서 AI ERP 도입이 막히는 지점이 바로 여기라고 생각해서, 이 부분에 가장 많은 설계 고민을 담았습니다.

## 15.8 5분 발표 스크립트 (전체본)

> 안녕하세요, ERPilot이라는 AI 기반 ERP SaaS 플랫폼을 소개하겠습니다.
>
> 중소기업 ERP 현장에서 가장 흔한 불편은 "어디서 뭘 찾아야 할지 모른다"는 것입니다. 재고 확인을 위해 메신저로 담당자를 찾거나, 연차 규정을 알기 위해 사내 위키를 뒤지는 식이죠. ERPilot은 이런 조회성 업무를 자연어 질문 하나로 해결하는 AI 업무 도우미를 ERP 코어에 결합했습니다.
>
> 기술적으로 가장 신경 쓴 부분은 AI 기능의 보안입니다. 단순히 LLM에게 DB 접근 권한을 주는 게 아니라, LLM이 호출하는 함수의 인자를 서버가 신뢰하지 않고, 실행 직전에 호출자의 실제 권한(Role, 부서, 본인 ID)으로 강제로 재작성하는 구조를 만들었습니다. 그래서 일반 직원이 AI에게 "팀장 급여 알려줘"라고 물어도, 시스템은 본인 데이터로 범위를 강제 한정하기 때문에 권한 우회가 구조적으로 불가능합니다.
>
> 데이터 조회 경로는 두 가지입니다. "이번 달 매출 요약해줘"처럼 정형 데이터 질문은 Function Calling으로 ERP DB를 직접 조회하고, "연차 규정 알려줘"처럼 비정형 문서 질문은 pgvector 기반 RAG로 사내 문서를 검색합니다. 이 둘을 1차로 저비용 모델이 분류한 뒤 메인 모델에 필요한 도구만 노출해서 비용과 정확도를 동시에 관리했습니다.
>
> 인프라는 NestJS와 PostgreSQL을 중심으로, Docker Compose로 컨테이너를 구성하고 Nginx로 리버스 프록시와 SSL을 처리했습니다. GitHub Actions로 테스트-빌드-EC2 배포까지 CI/CD 파이프라인을 직접 구축해서, 포트폴리오가 로컬에서만 동작하는 게 아니라 실제 운영 가능한 형태로 배포되어 있다는 점을 보여드리고 싶었습니다.
>
> 결론적으로 ERPilot은 "AI를 붙인 ERP"가 아니라, "ERP의 권한 모델을 AI 계층까지 일관되게 확장한 시스템"이라는 점이 가장 차별화되는 포인트라고 생각합니다.
